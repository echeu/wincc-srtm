     JCOP Framework version 9.3.0 - Release Notes - 26 November 2024
========================================================================

Requirements
------------
* Component Installation Tool 9.3.0 (or higher).

* WinCC OA 3.19 with patch P014 (CERN release 20241125) or higher. 
  NOTE: component installation or upgrade may fail on systems which
  do not have the adequate WinCC OA 3.19 patch installed!

* The fwConfigurationDB component requires an upgrade of its database
  schema to address an important (even though quite rare) issue with
  recipe consistency (issue FWCDB-1226, already fixed in the
  fwConfigurationDB-9.1.1 release).

Distribution package version number
-----------------------------------
Note that due to internal technical reasons the distribution 
package holds the "9.3.0.1" version number (ie. additional ".1"),
so that to incorporate a fixup for an issue spotted shortly after
the pre-release of the 9.3.0 version of the framework.



Resolved issues
-----------------

The complete list of resolved issues may be found at 
https://its.cern.ch/jira/issues/?filter=44527

Release notes:
--------------
* This yearly release of the JCOP Framework accumulates the releases of 
  individual components prepared since the 9.1.0 release in Autumn'2023
* The newest WinCC OA patches contain the completely new release of
  CtrlRDBAccess-11.03; it is a 100% compatible drop-in replacement,
  completely re-engineered and consolidated.
* As the new CtrlRDBAccess provides improved mechanism to keep the 
  User Interface responsive even during length operation, this may
  lead to a situation when a user may interact with the UI in a way
  that has previously not been possible (e.g. initiate another DB
  command while the previous one is still in progress).
  To address that, the fwConfigurationDB has a new option in the config
  file, fwConfigurationDB_ForceSynchronousMode (disabled by default)
  that would switch to synchronous (blocking) mode of DB operation,
  ie. the previous behaviour. Even though we consider this should not
  be needed, it stays as an optional additional compatibility switch.
* ConfigurationDB tool issue with recipe-data inconsistency (FWCDB-1226) 
  resolved, and a recipe-repair tool panel is distributed. 
* Advanced Recipe Operation panel in fwConfigurationDB tool now allows 
  to remove entries (devices, elements of a device) when editing the 
  recipeObject content.
* Numerous panels in fwConfigurationDB and fwDevice have been layouted
  (they adjust their content in response to being resized) and many of
  them got minor touch-ups to make them look correctly with the new
  color themes mentioned above.
* There are numerous improvements for the NextGen Alarm Screen,
  documented in the release notes of the component (over 30 issues
  resolved). Notably, filter-views and popups as well as the issues
  with fwAccessControl integration have been addressed.
* The New Trending Tool (fwTrending3 component) provides numerous
  enhancements to usability and functionality as well as bug fixes,
  and in particular mass configuration and interactive legend.
  Please, refer to the release notes of fwTrending3 for details.
  It makes use of the new visualization of the Trend Tree,
  enabling drag-and-drop as well as numerous new options in its
  context-menu. Numerous scalability/performance issues with the Trend Tree 
  used by the new Trending present in the previous releases have been addressed,
  notably for large dist systems.
* It is now possible to configure delayed alarms (ie. "time hysteresis")
  for the alarm configs. The feature is exposed in the fwAlertConfig
  panels and also integrated with the fwConfigurationDB tool.
* A number of issues related to the use of discrete alarms and 
  alarm status bit in the fwConfigurationDB and fwAlertConfig have been fixed.
* Significnt performance improvements in the framework device creation 
  functionality. Reference tests indicate up to 40% speed-up when
  creating fully populated CAEN crate with 16-boards.
  This is achieved by numerous improvements in the fwDevice and fwConfigs
  subcomponents of the Core.
* The new fwFSM version, 35, implements the long-awaited "FOR" loop construct.
* The documentation of the APIs for all components have been converted from HTML
  files to the Qt help, resulting in much smaller distribution package. 
  In addition, the documentation of all subcomponents of FwCore is now 
  centralized in a single help file (minimizing the number of entries at the 
  top of the documentation tree and making the installation faster).
* fwAccessControl component addresses the issue with serious performance 
  regression encountered in WinCC OA 3.19. It als resolves the problem with LDAP
  authentication not falling-back properly to local authentication when the 
  LDAP server cannot be contacted
* fwCAEN supports 3 new board types: A6601, A2551, A7435D
  and fixes issues with addressing of specific EASY boards: A1396, A380x family.
* fwCaen panels reworked to display the artwork (pictures of crates, boards) 
  only if fwCompatArtwok component is installed.
* fwCaen deprecates the A1676 branch controller configuration panels:
    fwCaen/fwCaenBoardSY1527A1676Import.pnl
    fwCaen/fwCaenBoardSY1527A1676ExpertConfiguration.pnl
    replaced with equivalent panels: fwCaen/fwCaenBranchControllerImport.pnl
  and
    fwCaen/fwCaenBranchControllerExpertConfiguration.pnl 
    (preparation for R6060 integration)
* fwELTEK component has operation panels now
* Majority of the components licensed on GNU Lesser General Public License
  (LGPL) v3, with dedicated release package (jcop-framework-oss).
  The OSS (Open Source Software) distribution does not include the 
  CERN-specific components (not being of general interest), notably 
  fwDIP, fwRack, fwCaV, fwMCS and it contains a dedicated build
  of the fwAccessControl component with CERN-specific elements removed.
* The artwork (icons, pictures) used by the components have been reviewed,
  and a set of dedicated new ones was developed, providing a homogeneous
  and modernized look. The new standard icons are placed in the new fwStdUi 
  subcomponent of fwCore, and are guaranteed to be compatible with Open Source
  licensing. All the Framework components have been reviewed and modified
  to use the new artwork and hence maintain the OSS license compliance
* To assure maximum compatibility for the existing project, the previous
  artwork elements are included in the new fwCompatArtwork component,
  which is a part of the CERN distribution of the Framework, but not the 
  OSS release. The components SHOULD be installed in the existing projects
  being upgraded to this release of the Framework, to guarantee that the 
  previously distributed artwork elements that might have been integrated
  into users' custom panels is still available (as it is NOT distributed
  anymore by the components to which they previously belonged). This way
  the compatibility is assured and there is NO NEED to modify any of the
  existing panels (even though it is recommended to consider using the
  new artwork delivered by fwStdUi instead for all the new developments
  and revisions of existing panels).
* The FwStdUi subcomponents provides a technical preview of the 
  "UI Theme swiching" functionality which allows, notably, to enable
  a dark color palette for the application. It has some limitation at the
  moment. The functionality is not active by default and may be enabled by
  uncommenting the line "fwStdUiThemesEnable = 1" in the project config file.
  When enabled, a theme-switch toolbar is added to GEDI, allowing to activate
  the "fwSdtUi-Light", "fwStdUi-Dark" themes or bring the UI back to the
  standard (drfault) theme. There are still some technical limitation 
  of this functionality in this release of the Framework, hence it should be
  considered a "technical preview" (demonstration of functionality) for the 
  time being.
* Associated with the theme-switching functionality, there are new color
  databases and themed sets of standard icons, with a proposal of color
  definitions that are provided (in the fwStdUi-Dark and fwStrUi-Light themes)
  for test and would serve for discussion of an update of the JCOP Framework
  Guidelines. Such discussion is planned to take place at the JCOP Framework
  Working Group forum in 2025.
* A set of new dialog standard boxes are also provided by fwStdUi, to be used
  efforlessly from CTRL code using the methods of the new FwStdUi class.
  They could be used, in particular, to ask a used for a confirmation, 
  an input of a value or a selection from a list. Look in the examples
  distributed with the fwStdUiUnitTests component for more details.
  At this moment the feature is considered to be a technical-preview,
  and used internally by e.g. the new Trending Tool component, hence it
  is not yet recommended for public use (and hence not documented).
  The feature will be presented for discussion at the JCOP Framework
  Working Group forum
* FwStdUi provides an initial proposal of feature-rich reference panels
  enabling rapid GUI development in particular when using Object-Oriented
  approach and CTRL++ (e.g. a table, a selection list, text line).
  Again, this is a technical preview of the feature, provided for testing
  with the aim of being discussed at the JCOP Framework Working Group forum.
  Examples of its use are distributed with the fwStdUiUnitTests component.
* The operation/configuration panels for the unDistributedControl components
  have been refreshed with a new look-and-feel (notably used by the upcoming
  release of the UNICOS framework).




     JCOP Framework version 9.1.0 - Release Notes - 31 October 2023
========================================================================

Highlights
----------
* First release of the new Trending component. For the time being it is kept as a separate
    (optional) component called "fwTrending3", which coexists with the standard fwTrending
    (and requires it to be installed). The new Trending provides a 1:1 functionality
    replacement for the standard one and also provides numerous long-awaited improvements
    (unlimited number of curves per plot, flexible layout of plots in a page, resizable
    layouted panels). It also implements some of the features of the "lbTrending" tool,
    (LHCb Trending) such as mass-creation of panels - these features will be developed
    further in upcoming releases of the new Trending component.
    New FwTrending blends with the standard one, with a switch between the two possible.
* AlarmScrenNG has all the requested and mandatory features implemented
* New component: fwStdLib regroups the recently implemented core functionality,
   notably enabling effective CTRL++ programming:
   - FwException class (previously in fwGeneral)
   - CtrlOOUtils (previously in fwCtrlUtils component)
   - CtrlJsonSerialization (previously in fwCtrlUtils)
* New component: fwStdUi providing utilities and building blocks for advanced user interfaces (panels):
   - new FwStdUi class with utility functions to pop up modernized dialog boxes
      for selection of item, text input or a message
   - new FwPanel class allowing to develope much cleaner code related to the use of reference panels,
      and binding shapes to class/struct, effectively enabling Model-View programming.
   - initial set of feature-rich Object-Oriented reference panels, working smoothly with CTRL++ lists
      and objects: selection list with filtering, powerful table with feature-rich headers, sorting/filtering.
      The API and look and feel not yet guaranteed to stay 100% compatible - a "feature preview".
   - definition of modernized color style definitions, for the "light" and "dark" style; initial set of
      standardinzed icons (will be enhanced in the upcoming releases)
   - initial support for "variable UI density" (e.g. changing the text/icon sizes and spacing at runtime, e.g.
      for more ergonomic use at large/high-DPI screens); initial integration in the new Trending Tree.
* FwTree component with the alternative Object-Oriented API for tree structure management and a new
   reference panel for visualization/operation, enabling drag-and-drop, etc; used e.g. in the new Trending Tool.
* Component help migrated to QtHelp format making the components much more compact, and integrating
   with WinCC OA help system. HTML help not distributed anymore, yet browsable at https://jcop.docs.cern.ch
   (latest but also a couple of previously releaased versions). Component postInstall files clean up the old
   HTML help files. The review and migration of panels' help has not yet been completed in this release - this
   cleanup will be continued in the subsequent releases of components and the framework
* All the binaries have their debug symbols stripped. The debug symbols may however be recovered from the
   distributions of individual components (PDB or .debug files)
* fwWiener: CAN->SNMP migration panel fully reviewed following users' feedback
* fwRack: Consolidation and improvements contributed by CMS, mainly:
   - Optimized calculation of devices powering state; added possibility to define device types directly powered
       from a backup source, that are calculated differently
   - Added possibility to optimize command queue w.r.t processing ACKA/ACK commands
   - Default location of ExcelFiles moved to data/fwRack/ to follow JCOP convention. Backwards compatibility
      with old directory structure assured. Fixed issues in default ExcelFile.csv
  More informations are provided in component release notes
* The new fwInstallation-9.1.0 pops up an information in any open GEDI window informing about the progress
   of postInstallation step execution.
* RDB Archiver component has a new functionality to convert password-encryption
* fwConfigs has a set of new utility classes to handle the S7 configuration datapoints
* All the components in this release have the version number set to "9.1.0" except for
   the fwFSM (v34r24) and fwDIM (v24r6) as well as the new fwTrending3, fwStdUi and fwStdLib components
* fw3DViewer component has been removed from the distribution and as of WinCC OA 3.19
    it will not be maintained anymore
* To have the NextGenArchiver configured consistently, notably with CERN-developed Oracle backend,
    additional components need to be installed. The next release of the Framework will include the
   "fwNextGenArchiver" component which is in development at the time of the release of the
   JCOP Framework 9.1.0. The component as well as detailed instructions will also be available on the 
   Framework Components downloads pages once released. In addition, the component providing the
   NGA Oracle Backend will be distributed internally at CERN, as it is not a part of the JCOP Framework;
   please contact the Industrial Controls Support teaam for more details.
   

Requirements
------------
* The latest version of fwInstallation tool 9.1.0 should be deployed in the project before
installing the JCOP Framework 9.1.0 components.
* We recommend the use of WinCC OA 3.19 with patch P006 or later.
* The framework is compatible with projects running either RAIMA or SQLite based data manager
* The framework is compatible with NextGen Archiver (with CERN Oracle backend), Oracle RDB Archiver
   and ValueArchiver (only available for RAIMA-based projects)
* Supported operating system versions are Alma Linux 9, RedHat Enterprise Linux 9 and Windows Server 2022
* For instruction concerning the installation/upgrade please refer to fwReadme.txt file in this distribution
* Note that the following warning reported in the project log file during the component upgrade are considered "normal"
  - while installing fwRack component, ASCII import issues warning with code 77 
    due to some new DP Elements being added to the _fwRackSettings datapoint in the new version of the component
  - while installing fwCaen component, ASCII import issues warning with code 55 when importing device
    definitions for CAEN Easy device types, the datapoints being created will have a different DPID allocated
    than the one specified in the DPL file - this is completely harmless in this context
  - while installing fwAnalogDigital component the same warning with code 55 is issued,
    which is again harmless
* During the upgtade of the nextGenAlarmScreen component, a verification is made to assure the configuration DPs
    are consistent. For older version of the component these datapoints have no version information and its
    migration may not be done in a reliable way; hence, during the installation, the user is asked if these datapoints
    should be dropped (which means loosing the settings contained in them).

     JCOP Framework version 9.0.0 - Release Notes - 4 April 2023
========================================================================

Initial, developer-grade release of the JCOP Framework for WinCC OA 3.19.
It is not intended to be used in production environment.

This release is feature-compatible with JCOP Framework 8.4.3 release.
The major exception to that is the fw3DViewer component that was removed
from the distribution (see deprecation notice below).
Minor enchancements and bugfixes not related with migration to WinCC OA
3.19 can be found in the 'List of resolved issues' section.

Components distributed in this release must be installed with
fwInstallation tool version 9.0.0 (or higher).

Requirements
------------
The 9.0.0 release requires WinCC OA 3.19 P002, which corresponds to the
CERN WinCC OA 3.19 release 20230403 (or later).

Supported Operating Systems are RHEL9/ALMA9, Windows 10, Windows 11,
Windows 2019 Server, Windows 2022 Server.

WinCC OA 3.16 is *NOT* supported.

The existitng projects must be upgraded to WinCC OA 3.19 prior to
installation of jcop-framework-9.0.0.

For more details about the procedure for installation or upgrade of
JCOP framework in a WinCC OA project please refer to the fwReadme.txt
file shipped with this release.

Deprecation of fw3DViewer component
-----------------------------------
The fw3DViewer component was removed from the distribution of the Framework
as deprecated. This component is not going to be migrated to WinCC OA 3.19.
A 3.16 version will remain available for download on the Framework pages.

fwAlarmHandling component entering maintenance mode
--------------------------------------------------
As of this release of the JCOP Framework, the fwAlarmHandling component
enters the maintenance mode. No active development is forseen for this
component and only bugfixes for blocker/critical issues will be provided.

It is highly recommended to use now the new Alarm Screen provided by the
fwAlarmScreenNg and AlarmScreenNg components instead. These components
reached the maturity state and are actively supported.

Binaries debug symbols availability
-----------------------------------
To optizimze the size of the distribution package all of the binaries got
the debug symbols separated-out. They are, however, included in the
distributions of individual components available to download on the
Framework pages.
Note that these files are not installed with component by default.

Known bug and limitations
-------------------------
The following non-critical issues are known and should be considered
as limitations of the current 9.0.0 developer version.
* FWAC-552    LDAP Connection timeout does not work
* FWAC-553    LDAP authentication: credentials-caching does not seem to work
* FWCORE-3570 fwAlarmHandling/fwAlarmScreen GroupConfiguration panels
              Import/Export does not work with SQLite DB

     JCOP Framework version 8.4.2 - Release Notes - 05 October 2021
========================================================================

This is the yearly release of the JCOP Framework for the currently recommended WinCC OA 3.16.

In addition to numerous bugfixes, it also brings the following new components

* fwLxiPs : integration of the LXI power supplies
* fwPdu   : integration of PDU (Power Distribution Unit) devices 
* fwWienerMPOD: integration of MPOD power supplies
* AlarmScreenNg and fwAlarmScreenNg: the new Alarm Screen
* fwOpcUaTools : tools used for OPC-UA migration

The complete list of changes as well as component-specific may be found below.
Users and developers are also encouraged to review the release notes of individual components.

The 8.4.2 release of the framework should be used with the latest version
of the fwInstallation tool, *8.4.3*, available at the Downloads page.
For existing projects we strongly recommend upgrading the fwInstallation prior
to the upgrade of components.

The recommended WinCC OA 3.16 patch version is P028, which corresponds
to the CERN release *20210804* available on the WinCC OA Service Downloas page.



New Alarm Screen
----------------
The new alarm screen is composed of two components: 
 - AlarmScreenNg : the base and generic implementation of the alarm screen
 - fwAlarmScreenNg: the integration and customizations of the generic implementation
    for the JCOP community.

For proper functioning, both of these components need to be installed (and upgraded).

The new alarm screen comes with an exhaustive user manual available at
https://edms.cern.ch/file/2607588/0.9.1/AlarmScreenNg-User_Manual-0.9.1.pdf

The new alarm screen has been validated with a battery of automated tests.
However, this release should still be used with caution (and treated as a "Preview" version)
in particular in production systems - please, use it alongside the "old" Alarm Screen.

The functionality of both the live and the historical mode should now be close to complete
and in a fully usable state. 
Please note that in order to use the historical mode, you need to use the Oracle RDB Archiver,
(or NextGenArchiver with the Oracle backend) and configure the AlarmScreenNg following the 
instructions in its documentaion.

WienerMPOD integration
----------------------
The fwWienerMPOD component extends the "Wiener" device tree in the DeviceEditorNavigator
with the possibility to declare new type of power supplies (ie. there is no separate
"vendor node" in the hardware view).

New Ctrl Extensions
-------------------
The fwCtrlUtils component have been extended with two new Ctrl Extensions
- CtrlJsonSerializer : robust and effective tool to serialize/deserialize CTRL++ objects
- CtrlOOUtils : tools to support object-oriented programming with CTRL++, such as
    extended RTTI, instantiation/inspection of objects, etc.

Both of these extensions are provided in a "preview" form - care and adequate testing should
be applied before relying on them in production systems. Nevertheless, we believe their features
should become available to developers already at this stage. Feedback on their usability
or suggested enhancement will be much appreciated. 
The JSON Setializer was developed to underpin the new Trending Tool (not yet released in this version),
yet the scope for its use is much wider (e.g. for effective calls of web services).
The CtrlOOUtils is the outcome of the JCOP CTRL++ workshop as well as discussions with ETM to
implement certain CTRL++ features requested by CERN users.
    
Trending
--------
This release of the framework contains the "classical" version of the fwTrending component,
with a couple of bug fixes and extensions. Notably, the trend plots and pages have been layouted, 
enhancing their functionality when being resized.
The "new" version of the Trending Tool has progressed over the past year, yet it is still considered
to be in a prototype stage, not suitable for being included in the framework distribution or applied
to production systems.

ConfigurationDB Tool
--------------------
The 8.4.3 version of the ConfigurationDB tool comes with the new DB Schema version, 
which addresses a potentially serious issues with data corruption while loading recipes.

For this reason it is strongly recommended to upgrade the ConfigurationDB schema straight
after the upgrade of the component.

UTF-8 CHARACTER ENCODING ENFORCED BY WINCC OA 3.16
--------------------------------------------------
The previously used "iso8859-1" character encoding for projects is
NOT SUPPORTED anymore by WinCC OA 3.16, and the use of the 
Unicode UTF-8 is enforced. This may pose compatibililty problems
with applications that make use of non-ASCII characters, such as
accented french characters, as their binary representaion in now
different. Even though existing projects are converted to UTF
during the upgrade process, some additional work may be required
to clean up possible artifacts. 
Some of the issues related to the ISO->UTF conversions which we
encountered and studied are documented at 
https://readthedocs.web.cern.ch/display/ICKB/Validation+WinCCOA+3.16

The code, panels, and data files of all the components of the framework
were reviewed and corrected for compatibility as necessary.

XML PANEL FORMAT
----------------
As a part of consolidation project, all the panels of the JCOP Framework
have been converted to the XML panel format, to ease their maintenance
(e.g. having more meaningful diff-files, etc). The file names for these
panels are conserved (.pnl), even though the content is now in XML.
There should be no difference in their use.
To convert own panels to the XML format, we recommend looking through
the procedure we list at
https://readthedocs.web.cern.ch/display/ICKB/Validation+WinCCOA+3.16

Nothe that upon the installation (or upgrade) of fwCore 8.4.0, the default
panel format used by GEDI whenever a panel is changed to XML.
This could be changed in the project config file (and such change will 
be respected during future upgrades of the fwCore component).

CLEANUP: LIBRARY LOADING
--------------------------
In the scope of the Consolidation&Redesign project, JCOP CB approved
the proposal to change the way in which CTRL libraries are loaded 
in the project.

The previous policy in which the libraries were loaded to all UI/CTRL
managers using the LoadCtrlLib statement in the config file is now
deprecated. Instead, all the necessary libraries must be explicitly
loaded throug the #uses statement, in whichever place they are used.

The change was applied to all the common components of the framework
WITH NOTABLE EXCEPTIONs of the large part of fwCore and fwInstallation
components that remain loaded "always".

This change has a direct impact on the projects that make use of the
CTRL API provided by any of the framework components: as of this release
of the framework it is mandatory to put explicit #uses statements for
necessary libraries for every panel or script or library that makes
use of component's functionality. The "fallback" solution of adding the
library-loading statement to the config file is technicaly possible,
yet STRONGLY DISCOURAGED by the JCOP CB.

To make the transition easier, it was decided that every component will
make sure it delivers a library called <fwCompName>/<fwCompName>.ctl, which
will be sufficient to load all the other libraries containing public APIs
of the component. Hence, for a user, it should be sufficient to put
the single #uses statement on this "main" library, and other libs will
be loaded (by subsequent #uses statements) as dependencies.
Cross-component dependencies are permitted (ie. the library of fwCompB
may put the #uses statement on the libs coming from fwCompA).

A mechanism is provided to have a fallback library containing
appropriate #uses statements for the main libraries from all installed component.
It is though as a "safety net": in case of dependency-resolution problems,
and emergency to resolve these in a production system it could be activated
through the LoadCtrlLib statement of the config file. This library,
is stored in <proj_dir>/scripts/libs/includes_legacy.ctl" file, and its
content is managed by the component installation tool, which regenerates
its content at every component installation/upgrade, therefore the file
should not be edited manually.
Whenever a maximum backward-compatibility is required, this file may be
made to be loaded by all the UI/CTRL managers in the project by putting
an appropriate LoadCtrlLib statement in the project config file.
Please refer to the comment on the top of this file for further information.


CLEANUP: STRUCTURE CHANGES
--------------------------

In response to the requirement of better modularity, the fwFSM
and fwDeviceEditorNavigator are *NOT* included in the fwCore anymore.
They become standalone tools that must be installed/upgraded explicitely.

This has a particular impact on the developer systems, as these tools 
were previously coming pre-installed by fwCore. We strongly recommend that
all developer system that need the functionality of either of these 
(or both) do remember to install them explicitly now.

The new fwTree component was separated-out from the fwFSM, and included
in fwCore; it provide the tree-handling functionality used by, among
others, fwDE&N, fwFSM, fwTree, without dependency of fwFSM.

A lot of consolidation effort was invested in cleaning up the structure
of components. Particularly, fwFSM and fwDIM components now have their
libraries kept in a subdirectory (e.g. libs/fwFSM rather than directly
in libs/). There should be no change for the users, however, because
the fwFSM component should take care of all the library-loading 
adjustments that were necessary. Proper cleanup of libraries that were
installed in previous location is done. 
The mechanism used to load the fwFsmUser.ctl (if exists) was changed too.
Previously used solution of creating a dummy empty file (if a user-provided
one did not exist) was replaced by a new mechanism provided by the fwGeneral
component of fwCore, whereby a library could be loaded conditionally, if it exists.
Ultimately, the handling of binaries was improved, with WCCOAdim and WCCOAsmi
managers having their own copies of the libdim.dll, which is properly loaded from
the appropriate subfolder. As the binaries now "know" where to look for libraries,
they do not need to be copied to the "bin/" folder directly and may remain in
a subfolder. As a consequence of that, the dns executables are now found in
bin/fwDIM and bin/fwFSM instead of the bin/ directly.

Similarly, the libraries of fwInstallation was moved to a subfolder;
as they are loaded by all the UI and CTRL managers, one should ensure
that there are no stray #uses "fwInstallation*.ctl" statements in the code,
as there is no need for it.

Help files and message-catalogue files were moved from the "iso88591"
subfolders to "utf8" subfolders, with proper cleanup.

A number of enhancements to fwInstallation allowed to simplify the
init/postInstall scripts; tasks such as proper installation of binaries,
cleanup of deprecated files and folders, etc are now handled much easier
(refer to the documentation of fwInstallation for details).

 - fwRPC,

To consolidate the new features provided by WinCC OA, this release provides

and recently integrated into the "HighSpeedProgramming" feature. Efforts were put
to enable mechanisms for project-specific help (rather than machine-specific, as
used by default by WinCC OA help files), with automatisation through the fwInstallation
tool. The new QtHelp-based help files are in stalled alongside the previously used HTML
help files, providing the same content (generated using the Doxygen tool), which integrates
smoothly into e.g. the GEDI script-editor. Care is taken to maintain the help collections

Numerous enhancements were introduced in the fwInstallation tool, providing

ENABLING NEW WINCC OA FEATURES
------------------------------
New features of CTRL++ scripting are now enabled by default when fwCore is installed.
This could be changed by editing the config file (modifications will be honoured
by future component upgrade). Even though none of these constructs is used as 
of the 8.4.0 release, it is likely they will be used in the very next releases already
for parts of the consolidation/redesign tasks (eg. in fwTrending).

The Script Editor of GEDI already contained a feature allowing to "parse" all the libraries
in the project, not only the ones that have already been loaded (and thus offer syntax-hinting, etc).
This mechanism had to be disabled until now, as it was delivering wrong hinting under certain circumstances
(e.g. parsing the files renamed to _old.ctl or _bak.ctl, etc). With this release of the framework
a mechanism of "whitelisting" was enabled through an additional file: config/config.gedi ; the
file whitelists explicitly some of the well-known libraries of the Framework, to make them "visible"
to the hinting-engine. This feature was of particular need because of a migration from the
central loading of libraries from the config file into explicit #uses.
The functionality is enabled by appending the "+config config.gedi" option to the UI manager(s)
configured in the project console with the "-m gedi" flag.

A dedicated tool to convert archive configs to the NextGenArchiver has been developed yet it
is not included in the Framework distribution - it could be obtained on demand from the
ICEControls.Support@CERN.CH .

The 8.3.0 release of the Framework marks the official support of

 - WinCC OA 3.16 + P005 (20181210) or later

The full list of resolved issues in Jira is available at
https://its.cern.ch/jira/issues/?filter=24373

CleanTheCode and deprecation of unused functions
---
In course of its development, the components of the framework
overwent a "CleanTheCode" operation which aimed at identifying
unused/unreferenced code pieces and deprecating them. During the process
we referred to the code of production DCS/ECS projects of the LHC experiments.
The fact of deprecation of function and panels should be transparent to the
cases where these might still be in use. Incidents of the use of deprecated
functionalities result in a printout of a message to the log (only once!),
and recording such incident, with a stack trace. A tool available in the 
"JCOP Framework" GEDI menu allows to list the incidents. 
In addition, deprecated reference panels, when used, will flash a "Deprecated" 
text for 3 seconds on opening.
Deprecated functions were moved to separate libraries (the name of which 
implies it), which is loaded from the libraries that were original locations
for deprecated construct; they are also listed in online help as deprecated.
To signal deprecation in the code, a new public function FWDEPRECATED() is provided.
It could also be used in custom code to take advantage of the functionality.

Cleaner Project Console and where is my DeviceEditor&Navigator
---
As of this version the installation of component will avoid adding 
dedicated UI managers, such as the one for DE&N to the project console.
Instead, these tools are now added to the "JCOP Framework" menu in GEDI.
Similarly, managers which are not supported on Linux (e.g. OPC-DA driver)
are not added to console. At the same time, for devices such as CAEN,
where both OPC-DA and OPC-UA are supported, both driver types will be added
to the console (if installed in non-interactive mode).

Cleaner installation of binaries
---
For certain binaries, such as fwAccessControl's CrtlLDAP extension,
or fw3DViewer EWO, only necessary binaries are copied during the installation.
A new <bin> tag in fwInstallation-8.3.0 allows for automation of this task.

OPC-UA and multiple types of addressing for devices
---
The Device-management panels as well as API of fwDevice were enhanced to 
support cases of devices that could have multiple types of addresses, such 
as CAEN. As a result, it is possible at device-creation time to specify
which type of addressing (eg. OPC-UA vs OPC-DA) should be applied.
Diagnostics inside fwDevice functions was also enhanced.

Layout-management (resizing) of panels
---
Some of frequently used panels, such as DeviceEditor&Navigator were redesigned
to allow resizing at runtime (using layour management). These panels are 
visually marked by 3 diagonal strips in their bottom-right corner.

FwCAEN component enhancement
---
OPC-UA addressing is fully supported by fwCaen component now, and a tool
(panel) to migrate addressing of existing devices from OPC-DA to OPC-UA
was provided. In addition, device definitions for all existing models were
reviewed and inconsistencies were eliminated.
A number of new board models was also integrated (see the component help
file for details).

New archive smoothing behaviour for status/user bit-changes
---
Archive smoothing behaviour was tuned so that a change in status bits
(such as invalid bit) and user bits (used e.g. for data quality) would
cause the value to be archived overpassing the smoothing criterias.
In particular for trend plots this would allow to spot incidents such
as hardware or driver disconnections for types of drivers that support
exposing such information through status bits.

Improvements to fwInstallation
---
Numerous improvements and bugfixes were added to the Installation Tool,
including the new <bin> tag for installation of binaries with no need
for postInstall scripts

Deprecation of fwRPC component
---
The fwRPC component was removed from the distribution of the Framework
as deprecated. It might be downloaded separately from the Framework pages.

New fwViewer component
---
A new fwViewer tool was added to distribution, allowing to open URL links
leading to HTML and PDF documents (provided one uses WinCC OA 3.16) and 
display them in a panel, rather than an external browser. This may 
allow for significantly faster opening of such documents and may remove
the necessity of having a dedicated PDF viewer installed on your runtime
system. In addition adequate mechanism to integrate logging it to 
CERN EDMS document handling system was integrated.
FwViewer replaces the functionality of fwWebBrowser panel previously 
available by the fwGeneral component.

    remote dist systems (FWCDB-1107); by default, a compatibility mode
    is kept in which remote aliases are not resolved (ie. a recipe
    containing them may not be applied); a new mode is activated by
    setting the global variable fwConfigurationDB_allowResolveRemote to
    true (to be effecive one also needs to set the optional parameter
